# Техническое задание — NavIS
**Проект:** NavIS — Навигатор Информационных Систем (Service Catalog / API Explorer)
**Версия:** 1.0
**Дата:** 2026-03-20

---

## Идея

Веб-приложение для SRE-инженеров и разработчиков, позволяющее:
- Выбрать ИС из списка
- Увидеть сервисы внутри
- Тапнуть на сервис → развернуть методы
- Тапнуть на метод → увидеть request/response, базы данных, хранимые процедуры
- Визуализировать граф зависимостей между сервисами, БД и процедурами

Приложение разворачивается на корпоративном Linux-сервере в Docker, работает внутри сети.
Конечный пользователь не вводит учётные данные — доступ через SSO.
Интеграции (Confluence, Git, БД) работают через сервисные аккаунты, настраиваемые администратором.

---

## 1. Цель и задачи проекта

- Создать внутренний веб‑сервис, позволяющий:
  - Просматривать каталог информационных систем (ИС), их сервисов и методов.
  - На карточке метода видеть: описание, request/response (из спецификаций), связанные БД и хранимые процедуры, зависимости.
  - Искать по ИС/сервисам/эндпоинтам/процедурам.
  - Визуализировать граф зависимостей между API, БД, процедурами и другими сервисами.
  - Добавлять новые ИС через подключение к источникам знаний (Confluence, Git, при необходимости — БД).
- Ключевые принципы:
  - Конечный пользователь не вводит личные учётные данные (SSO).
  - Интеграции работают через сервисные учётные записи, настраиваемые администратором.
  - Приложение развёртывается на одном корпоративном Linux‑сервере в Docker.

---

## 2. Область применения и ограничения

- Внутренняя сеть компании, доступ с корпоративных рабочих мест (Windows, браузер Chrome/Edge).
- Данные — только служебные артефакты: спецификации API, схемы, SQL/DDL/DML, описание ИС из Confluence, метаданные БД (read‑only).
- Не входит в объём:
  - Транзакционные данные продакшн‑БД.
  - Активное тестирование/вызов API.
  - Автоматическая генерация диаграмм из проприетарных инструментов без API.

---

## 3. Пользовательские роли и доступ

- **Гость** (optional): просмотр каталога без детализации (если разрешено политикой).
- **Viewer** (по умолчанию): просмотр всех данных «только чтение».
- **Editor**: ручной мэппинг/коррекция связей, добавление/редактирование метаданных ИС.
- **Admin**: управление источниками, сервисными учетными данными, расписаниями, правами.
- **Аутентификация:** корпоративный SSO (Kerberos/NTLM через обратный прокси или SAML/OIDC через корпоративный IdP/Keycloak).
- **Авторизация:** роль‑based (RBAC), опционально привязка видимости ИС к командам/отделам.

## 4. Основные пользовательские сценарии

### 4.1. Обзор каталога
- Пользователь открывает веб‑приложение (host:port → далее корпоративное DNS).
- Видит список ИС (поиск/фильтры: владелец, метки, среда).
- Клик по ИС → список сервисов.
- Клик по сервису → список интерфейсов/API.
- Клик по интерфейсу → методы (эндпоинты). Отображаются метод, путь, протокол, версия.

### 4.2. Карточка метода
- Поля:
  - Общие: имя, путь/операция (GET/POST/…), протокол (HTTP/gRPC), версия API, владелец/команда, среда.
  - Request: схема, обязательные поля, примеры.
  - Response: коды, схемы, примеры.
  - Связи: какие БД используются, таблицы/процедуры, другие сервисы (внешние вызовы).
  - Источники (evidence): ссылки на Confluence, Git‑файлы/коммиты/теги, версии спецификаций.
- Действия:
  - Открыть первоисточник (в новой вкладке).
  - Посмотреть граф связей от метода до БД/процедур.

### 4.3. Поиск
- Полнотекстовый поиск по: ИС, сервисам, интерфейсам, методам, путям, таблицам, процедурам, репозиториям.
- Подсказки/автодополнение.
- Фильтры: тип сущности, среда (dev/test/prod), язык/протокол, наличие связей с БД.

### 4.4. Граф зависимостей
- Интерактивный граф: узлы (ИС, сервис, интерфейс, метод, внешний сервис), рёбра (типизированные).
- Навигация: приближение/отдаление, подсветка путей, фильтрация по типам узлов/рёбер (кнопки-тоглы).
- Узлы типа `external` — сервисы, упомянутые в архитектурных диаграммах, но не зарегистрированные в каталоге.

### 4.5. Добавление новой ИС
- Editor/Admin открывает «Добавить ИС».
- Вводит базовые метаданные: название, краткое описание, владелец/ответственные, теги.
- Указывает источники:
  - Confluence: Space/страницы/фильтр по заголовку.
  - Git: группа/репозитории/ветки/путь к спецификациям.
  - БД (опционально): выбор из преднастроенных подключений read‑only.
- Сохраняет. Worker запускает сбор/парсинг. По завершении виден результат с логом.

### 4.6. Ручной мэппинг/коррекция (Editor)
- Привязка метода к конкретной процедуре/таблице при отсутствии авто‑детекта.
- Редактор метаданных (команда, SLA, контакт).
- История изменений и откат.

## 5. Функциональные требования

### 5.1. Интеграции/коннекторы

**Confluence (реализовано):**
- Тип сервера: Confluence Server / Data Center (on-premise).
- Авторизация: Basic auth (`username:password`). Планируется поддержка PAT.
- Сбор: перебор страниц space с пагинацией, поиск вложений с `mediaType: application/vnd.jgraph.mxfile`.
- Парсинг draw.io диаграмм:
  - Формат файла: `<mxfile>` (plain XML или base64+deflate сжатый `<diagram>`).
  - Извлечение сущностей: верхнеуровневые вершины (vertex).
  - Извлечение связей: рёбра с label (протокол/направление).
  - Сопоставление с каталогом: exact → case-insensitive → partial match по имени сервиса.
  - Несопоставленные → узлы `external` в графе.
- Фильтр страниц по заголовку (опционально).

**Git (реализовано):**
- Провайдеры: GitHub REST API v3, GitLab REST API v4, Bitbucket Server REST API 1.0.
- Авторизация: PAT/токен (опционально).
- Парсинг: OpenAPI/Swagger YAML и JSON → Service → Interface → Method с дедупликацией.
- Фильтр путей: glob-шаблон (например, `api/**/*.yaml`).

**БД (запланировано):**
- Подключение read‑only (через заранее заведённые сервисные аккаунты).
- Поддерживаемые СУБД:
  - **MS SQL Server** (приоритет 1) — основное корпоративное хранилище (WBase, BackOffice, FinamEDO и др.), порт 1433
  - **PostgreSQL** — используется рядом сервисов, включая NavIS
  - **ClickHouse** — аналитическое хранилище
- Инвентаризация: схемы, таблицы, представления, процедуры/функции, зависимости (references).
- Парсинг текста процедур для выделения обращений к таблицам и внешним вызовам (по шаблонам).
- Альтернатива: импорт подготовленных вью/дампов от DBA, если прямой доступ невозможен.

**Планировщик (запланировано):**
- Расписания сбора (настраиваемые per‑источник).
- Инкрементальные обновления.
- Ретраи и DLQ для проблемных артефактов.

### 5.2. Нормализация и консолидация
- Приведение артефактов к общей модели (ИС → Сервис → Интерфейс → Метод).
- Дедупликация по ключам (имя/путь/версия/репозиторий).
- Хранение evidence — список первоисточников каждого факта.

### 5.3. API backend
- REST для UI.
- Документация: `/api/docs` (Swagger UI), `/api/redoc`.
- Подробное описание — см. `API.md`.

### 5.4. UI
- SPA (современный браузер, без плагинов).
- Тёмная/светлая тема (CSS variables, переключатель, localStorage).
- Навигация по дереву ИС → Сервис → Интерфейс → Метод.
- Карточки сущностей с разворачиваемыми деталями.
- Глобальный поиск с кликабельными результатами.
- Граф связей (Cytoscape.js): зум, перетаскивание, фильтры по типам узлов.
- Страница источников: добавление Git и Confluence, запуск импорта, история джобов с логом.

## 6. Нефункциональные требования

- **Производительность:**
  - Поиск: <1 сек при индексе до 1 млн сущностей.
  - Открытие карточки: <1 сек.
  - Построение графа до 200 узлов: <2 сек.
- **Масштабируемость:** один сервер; возможность разделить ingest/search при росте.
- **Доступность:** 99% в рабочие часы.
- **Безопасность:** SSO, RBAC, секреты в хранилище, маскирование чувствительных данных, аудит-лог.
- **Логирование:** уровни info/warn/error; метрики времени парсинга и ошибок интеграций.
- **Совместимость:** Chrome/Edge последних 2 версий; Linux x86_64, Docker 20+, docker‑compose v2+.
- **Язык интерфейса:** RU (в дальнейшем EN).

## 7. Архитектура и развёртывание

### 7.1. Топология (Docker Compose на одном сервере)

```
┌─────────────────────────────────────────────────┐
│                 Корпоративный сервер             │
│                                                 │
│  Nginx (reverse proxy, TLS, SSO)                │
│       │                                         │
│  ┌────▼──────┐   ┌──────────────┐              │
│  │ Frontend  │   │   Backend    │              │
│  │  React +  │◄──│   FastAPI    │              │
│  │Cytoscape  │   │   (Python)   │              │
│  └───────────┘   └──────┬───────┘              │
│                         │                      │
│              ┌──────────┼──────────┐           │
│              ▼          ▼          ▼           │
│         PostgreSQL    Redis     Worker          │
│         (основная БД) (кэш/     (Ingest)       │
│                        очередь)                │
└─────────────────────────────────────────────────┘
```

Сервисы Docker Compose: `nginx`, `frontend`, `backend`, `worker`, `postgres`, `redis`

### 7.2. Модель данных (основные сущности)

| Сущность | Ключевые поля |
|----------|---------------|
| System (ИС) | id, name, description, owner, tags, environments |
| Service | id, system_id, name, description |
| Interface (API) | id, service_id, type (http/grpc), version, spec_ref |
| Method (Endpoint) | id, interface_id, name, http_method, path, request_schema, response_schema |
| Edge | id, from_id, from_type, to_id, to_type, kind, confidence |
| IngestSource | id, system_id, type (git/confluence), provider, repo_url, confluence_url, space_key, token, branch, path_filter |
| IngestJob | id, source_id, status, files_found, methods_created, log, error, started_at, finished_at |
| Source/Evidence | id, method_id, type, ref, hash, collected_at |

**Типы рёбер (Edge.kind):** `contains`, `calls`, `depends-on`, значения из label draw.io диаграмм

**Типы узлов графа:** `system`, `service`, `interface`, `method`, `external`

### 7.3. Хранилище и индексы
- PostgreSQL: индексы по name/path, полнотекстовые индексы, индексы рёбер по from/to/kind.
- Redis: очередь инgest-задач (BLPOP).

## 8. Процессы интеграции

Конвейер: **Crawl → Parse → Normalize → Reconcile → Link → Index**

- Git: поллинг → парсер OpenAPI → Service/Interface/Method с дедупликацией.
- Confluence: перебор страниц → скачивание draw.io вложений → парсинг XML → рёбра графа.
- БД (запланировано): обновление по расписанию → инвентаризация схем/процедур.

## 10. Безопасность

- SSO — пользователи не вводят пароли в приложении.
- Сервисные аккаунты для Confluence/Git/БД — только у Admin.
- Секреты хранятся в БД; рекомендуется перевод на Docker secrets / Vault.
- Маскирование чувствительных полей payload (токены, email, телефоны).
- Аудит-лог: входы, изменения мэппингов, запуски сборок.
- Исключение из индексации конфиденциальных страниц/файлов (по меткам/ACL).

---

## 11. Критерии приёмки

- Дерево ИС → Метод работает, время отклика <1 сек на карточку.
- Карточка метода: request/response + источники.
- Поиск <1 сек при 200к сущностей.
- Граф: узлы всех типов (включая external), фильтры по типам, глубина 1–3.
- Добавление ИС через Confluence (draw.io → рёбра) и Git (OpenAPI → методы).
- Логи импорта видны в UI.
- Ручное добавление сервисов/интерфейсов/методов.

## 13. Риски

| Риск | Мера снижения |
|------|---------------|
| Недостаточные права источников | Согласовать сервисные аккаунты заранее; fallback — импорт дампов |

## 15. Глоссарий

| Сервис | Компонент ИС, предоставляющий API |
| Интерфейс | Конкретное API (например, HTTP v1) |
| Метод | Эндпоинт/операция API |
| Коннектор | Модуль интеграции с внешним источником |
| Ингест | Процесс сбора/парсинга/загрузки артефактов |
| Evidence | Ссылки на первоисточники фактов |
| External | Внешний сервис из диаграммы, не зарегистрированный в каталоге |
